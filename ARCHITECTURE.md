## Component Architecture

### 1. Registry Manager (`lib/registry.js`)

**Purpose:** Central project registry management

**Responsibilities:**
- Load/save registry from `~/.dpm/registry.json`
- Add/remove projects
- Query projects by name, port, or host
- Initialize new registry
- Export registry for external tools

**Key Methods:**

```javascript
// Load registry from disk
async load()

// Save registry to disk
async save()

// Initialize empty registry
async initialize()

// Add project to registry
addProject(projectName, port, host)

// Remove project from registry
removeProject(projectName)

// List all projects (with optional filter)
listProjects(filter)

// Get all ports in use
getAllPorts()

// Find project by port
getProjectByPort(port)
```

**Data Structure:**

```json
{
  "_metadata": { /* version, timestamp */ },
  "_reserved": { /* port allocation guidelines */ },
  "project-name": {
    "port": 9100,
    "host": "project-name.localtest.me",
    "addedAt": "2024-03-21T10:30:00.000Z"
  }
}
```

### 2. Port Manager (`lib/port-manager.js`)

**Purpose:** Intelligent port allocation and conflict detection

**Responsibilities:**
- Auto-assign next available port
- Detect port conflicts
- Check if port is in use by system process
- Organize ports into ranges
- Provide port statistics

**Port Range Strategy:**

```
9000–9099  → Core Infrastructure
            (Jenkins, Jira, Artifactory, Consul, etc)

9100–9499  → Project Services
            (APIs, microservices, workers)

9500–9999  → Experimental & Docker
            (Test services, temporary projects)
```

**Key Methods:**

```javascript
// Get next available port
async getNextAvailablePort(preferredBlock)

// Check if port already assigned in registry
findConflict(port)

// Check if port in use by OS
async isPortInUse(port)

// Validate port is in valid range
validatePortRange(port)

// Get stats on port usage
getPortStats()

// Get port range definitions
getPortRanges()
```

### 3. Hosts Manager (`lib/hosts-manager.js`)

**Purpose:** Manage system `/etc/hosts` file

**Responsibilities:**
- Read/write `/etc/hosts` entries
- Create backups before modifications
- Mark DPM-managed entries with identifiers
- Support platform-specific paths (Windows/Unix)
- Provide alternative approaches (localtest.me)

**Key Methods:**

```javascript
// Update /etc/hosts with registry entries
async updateHosts(entries, removeStale)

// Create backup of hosts file
async backup()

// Remove all DPM entries
async removeAllDPMEntries()

// Verify entry exists
async verifyEntry(host)

// Get path to hosts file
getHostsPath()
```

**Entry Format in /etc/hosts:**

```
# === DEV PORT MANAGER START ===
# Auto-generated by Dev Port Manager
# Updated: 2024-03-21T10:30:00.000Z
127.0.0.1 my-service.localtest.me # my-service (port 9100)
127.0.0.1 api-gateway.localtest.me # api-gateway (port 9101)
# === DEV PORT MANAGER END ===
```

### 4. Config Generator (`lib/config-generator.js`)

**Purpose:** Generate per-project configuration files

**Responsibilities:**
- Generate `.project-dev.json`
- Inject port forwarding into `.vscode/settings.json`
- Generate `.env` for docker-compose
- Create bootstrap scripts
- Generate project documentation

**Key Methods:**

```javascript
// Generate project config object
generateProjectConfig(projectName, port, host)

// Write .project-dev.json
async writeProjectConfig(outputDir, config)

// Generate VS Code settings
generateVSCodeSettings(projectPort)

// Write/merge .vscode/settings.json
async writeVSCodeSettings(outputDir, projectPort)

// Generate docker-compose .env
generateDockerEnv(projectName, port, host)

// Write .env (append if exists)
async writeDockerEnv(outputDir, projectName, port, host)

// Generate bootstrap script
generateBootstrapScript(projectName, outputDir)

// Write bootstrap scripts (bash + powershell)
async writeBootstrapScripts(outputDir, projectName)

// Generate project README
generateReadme(projectConfig)

// Write PROJECT_DEV.md
async writeReadme(outputDir, projectConfig)
```

### 5. CLI Entry Point (`bin/dpm.js`)

**Purpose:** Command-line interface for users

**Responsibilities:**
- Parse and route CLI commands
- Orchestrate registry, port manager, hosts manager
- Handle errors and provide user feedback
- Format output (tables, JSON, text)

**Commands:**

| Command | Purpose |
|---------|---------|
| `assign` | Assign port to project |
| `list` | List all projects |
| `sync` | Update /etc/hosts |
| `remove` | Remove project |
| `init` | Initialize registry |
| `status` | Show system status |

## Data Flow

### Flow 1: Assign Port

```
User Input
    ↓
dpm assign <project> [--options]
    ↓
Creates Registry + PortManager
    ↓
Load existing registry
    ↓
Is manual port specified?
    ├─ Yes → Validate no conflict
    └─ No → Get next available port
    ↓
Add to registry
    ↓
Save registry
    ↓
ConfigGenerator creates:
    ├─ .project-dev.json
    ├─ .vscode/settings.json (if --vscode)
    └─ .env (if --docker)
    ↓
Output success + details
```

### Flow 2: Sync /etc/hosts

```
User Input
    ↓
sudo dpm sync
    ↓
Load registry
    ↓
Get HostsManager
    ↓
Backup `/etc/hosts`
    ↓
Extract project entries from registry
    ↓
Remove old DPM entries from /etc/hosts
    ↓
Insert new DPM entries
    ↓
Save file (with elevated privileges)
    ↓
Output result
```

### Flow 3: List Projects

```
User Input
    ↓
dpm list [filter]
    ↓
Load registry
    ↓
Apply filter (substring match)
    ↓
Format output:
    ├─ Table (default)
    └─ JSON (--json flag)
    ↓
Display to user
```

## Port Allocation Algorithm

```
GetNextAvailablePort(preferredBlock='project-dev'):
  1. Load all used ports from registry
  2. Load port range for preferredBlock
  3. For each port in range:
     a. Check if in registry
     b. Check if in-use by OS (netstat/lsof)
     c. Return first available
  4. If block full, try other blocks
  5. If all full, throw error
  6. Return port
```

## Conflict Detection

```
ValidatePort(port):
  1. Check if port in registry
     → If yes, return conflict project name
  2. Check if port in use by OS
     → If yes, return "system process"
  3. If no conflicts, return success
```

## Error Handling

All components implement error handling:

```javascript
try {
  // Operation
} catch (error) {
  throw new Error(`Descriptive error: ${error.message}`);
}
```

CLI catches errors and displays to user with context:

```
✗ Error: Port 9100 already assigned to project: api-gateway
```

## Security Considerations

1. **Registry File**: `~/.dpm/registry.json`
   - Readable by default (no secrets)
   - Can be synced to team shared storage
   - Should be gitignored per-project

2. **Port Access**: Ports 9000-9999
   - Localhost only (127.0.0.1)
   - Not exposed to network by default
   - User responsible for firewall rules

3. **/etc/hosts Access**:
   - Requires sudo/admin privileges
   - Changes are localized to machine
   - Backups created automatically

4. **Project Configs**:
   - `.project-dev.json` contains non-sensitive data
   - Can be committed to VCS
   - Secrets should be in `.env` (gitignored)
